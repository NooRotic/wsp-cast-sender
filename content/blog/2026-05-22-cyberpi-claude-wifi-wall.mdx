The Makeblock CyberPi is a handheld microcontroller learning device with a color screen, microphone, buttons, a gyroscope, a speaker, and a built-in WiFi module (the ESP3200). You program it in Python over USB through Makeblock's mBlock editor. It is aimed at classrooms.

For a project I had in mind, it looked perfect. The CyberPi would be a pocket-sized Claude remote: press a button, speak a prompt into the mic, the device uploads the audio over WiFi to a small server I'd already built, the server hits the Claude API, and the response comes back over the speaker. A standalone always-with-me AI companion roughly the size of a credit card with a screen on top.

I bought the device, plugged it in, started writing Python. Then I hit the wall.

## What I expected the WiFi to do

The ESP3200 is right there on the board. It is advertised in the documentation. The mBlock visual blocks include WiFi connect, HTTP request, and "send to IoT cloud" actions. From the surface it looks like a normal connected microcontroller.

So my mental model was: import the WiFi module from Python, call connect, then either do an HTTP POST or open a socket. Standard stuff. The ESP32 family supports all of that natively when you have direct access. I assumed CyberPi would expose the same.

## Why this is by design (probably)

The CyberPi is for classrooms. Schools want predictable, sandboxed devices that connect only to vetted services, do not allow arbitrary outbound HTTP, and cannot accidentally exfiltrate audio recordings of children to an arbitrary endpoint. From that angle, locking the WiFi behind a firmware-managed cloud bridge is the right call.

It is just not the call that lets you build a personal Claude remote.

This is one of those moments where hobbyist intent and educational-product design point in different directions. The same hardware that would unlock the project I had in mind is what makes it inappropriate for the classroom market the device exists to serve. The lock is not malice, it is product strategy. It still kills my project.

## The lesson for next time

When evaluating a piece of hardware for an open-ended project, the first question I will ask now is "does this advertised module have an open driver path from arbitrary user code, or is it gated by the vendor's bundled SDK?" The ESP3200 module exists on the CyberPi. The application-layer API to it does not. Knowing which side of that line a module lives on is the difference between a weekend project and a wasted purchase.

The CyberPi is going back in the drawer for now. Maybe a kid in my life ends up using it for what it was built for. That is fine. The lesson is mine.
